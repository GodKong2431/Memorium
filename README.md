# Memorium

`Memorium`은 방치형 액션 핵앤슬래쉬 구조를 목표로 한 Unity 프로젝트입니다. 현재 프로젝트는 `Addressables + CSV` 기반 데이터 로딩, `JSON` 기반 분리 저장, 모듈형 인벤토리, 층 단위 스테이지 진행과 무한 맵 배치를 중심으로 구성되어 있습니다.




























## 기술 스택

- Unity `6000.3.7f1`
- URP `17.3.0`
- Addressables `2.8.1`
- Input System `1.18.0`
- AI Navigation `2.0.9`
- JSON Save / Object Pool / Scene Transition

## 핵심 기술 소개

### 1. 테이블 기반 데이터 로딩

- `DataManager.cs` : Addressables 라벨 `CSV_Data` 기준으로 CSV 테이블을 비동기 로드하고, 재시도·타임아웃·상태 메시지까지 포함한 초기 데이터 파이프라인을 담당합니다.
- `StageKeyCatalog.cs` : `StageManageDict`를 `StageType` 단위로 캐싱해, 요청된 스테이지 타입과 레벨을 실제 스테이지 데이터로 빠르게 해석합니다.
- `RewardManager.cs` : 드롭 테이블과 던전 보상 테이블을 바탕으로 전투 보상, 던전 미리보기, 클리어 보상 지급 흐름을 데이터 기반으로 관리합니다.

### 2. 스테이지 진행과 맵 전개

- `StageManager.cs` : 스테이지 정보 로드, 현재 스테이지 상태 전환, 킬 카운트 기반 진행도 계산, 보스 소환 조건, 클리어/실패 저장을 함께 관리합니다.
- `MapManager.cs` : 층별 맵 프리팹 그룹과 BGM을 `StageType` 기준으로 선택하고, 현재 층의 맵 조각과 위치 정보를 런타임에 구성합니다.
- `InfinityMap.cs` : 징검다리 형태의 무한 맵을 재배치하고, 목표 지점과 플레이어 이동 경로를 다음 맵 조각 기준으로 갱신합니다.
- `MonsterSpawner.cs` : 맵 조각별 소환 트리거 위치를 이동시키고, 현재 스테이지 테이블에 맞는 일반 몬스터 또는 보스를 오브젝트 풀에서 꺼내 배치합니다.

### 3. 인벤토리 구조

- `InventoryManager.cs` : 아이템 ID를 `ItemType`으로 해석해 각 도메인 모듈로 라우팅하고, 아이템 수량 변경 이벤트를 공통 경로로 발행합니다.
  - `CurrencyInventoryModule.cs`
  - `EquipmentInventoryModule.cs`
  - `SkillInventoryModule.cs`
  - `PixieInventoryModule.cs`
  - `GemInventoryModule.cs`
  - `StackItemInventoryModule.cs`

### 4. 데이터 저장 구조

- `AutoDataSaveManager.cs` : 각 시스템의 `SaveData` 객체를 수집한 뒤, 주기 저장과 앱 일시정지·종료 시 저장을 수행하며 `Dirty` 상태만 골라 저장합니다.
- `JSONService.cs` : 동기/비동기 JSON 저장, 임시 파일 작성 후 교체, 제네릭 로드/삭제를 제공하는 공용 저장 서비스입니다.
- `SaveData.cs` 계열 : 기능별 저장 데이터 구조를 분리해 각 시스템이 독립적으로 세이브/로드를 관리할 수 있도록 구성합니다.
  - `SaveAbilityStoneData.cs`
  - `SaveBingoData.cs`
  - `SaveCurrencyData.cs`
  - `SaveEquipmentData.cs`
  - `SaveGachaData.cs`
  - `SaveGemData.cs`
  - `SavePixieData.cs`
  - `SavePlayerData.cs`
  - `SaveQuestData.cs`
  - `SaveSkillData.cs`
  - `SaveStageData.cs`

### 5. 장비 성장과 자동화

- `EquipmentHandler.cs` : 장비 자동 장착, 자동 합성, 강화 연동, 장비 UI 갱신 요청을 관리하는 장비 시스템의 진입점입니다.
- `PlayerEquipment.cs` : 현재 장착 중인 장비 테이블 데이터를 보관하고, 장착 변경 시 스탯 슬롯과 저장 데이터를 함께 갱신합니다.
- `ReinforecementEquipmentStat.cs` : 강화 수치에 따라 장비별 기본 스탯과 보너스 스탯 증가량을 계산해 캐싱합니다.

### 6. UI와 씬 전환 프레임워크

- `UIControllerBase.cs` : `Initialize -> Subscribe -> RefreshView -> Unsubscribe` 흐름으로 UI 생명주기를 통일한 공통 베이스 클래스입니다.
- `SceneController.cs` : 비동기 씬 로드와 페이드/아이리스 전환을 묶어, 씬 준비 완료 시점까지 포함한 화면 전환 흐름을 관리합니다.

### 7. 런타임 최적화

- `ObjectPoolManager.cs` : 몬스터, 이펙트, 기타 프리팹을 풀 단위로 재사용해 런타임 생성/파괴 비용을 줄이는 공용 오브젝트 풀입니다.









## 기술 결정 기록

장비 성장과 자동화는 기능 설명 성격이 강하다고 판단해 기술 결정 기록에서는 제외했습니다. 아래 항목은 현재 프로젝트의 핵심 구조를 왜 채택했는지에 대한 기록입니다.

### ADR-001. 테이블 기반 데이터 로딩

- 배경 : 기획자와 주고받아야 하는 데이터 양이 많고, 데이터를 수정·정리·검수하는 과정이 자주 발생했습니다.
- 결정 : `Addressables + CSV` 기반의 테이블 로딩 구조를 채택했습니다.
- 채택 이유 : 테이블 기반 구조는 대량 데이터를 정리하고 전달하기 쉽고, 기획자와 협업할 때도 관리가 편합니다. 또한 데이터 추가와 수정이 반복되는 프로젝트 특성상 테이블 단위 관리가 유지보수에 유리하다고 판단했습니다.

### ADR-002. 스테이지 진행과 맵 전개

- 배경 : 기획 의도상 플레이어가 끝없이 앞으로 달리는 느낌의 스테이지를 구현해야 했습니다.
- 결정 : 긴 맵을 한 번에 배치하는 대신, 맵 청크를 뒤에서 앞으로 한 칸씩 재배치하는 방식의 무한 맵 구조를 채택했습니다.
- 채택 이유 : 맵 전체를 길게 만드는 방식보다 리소스 관리가 효율적이고, 적은 맵 조각으로도 무한히 이어지는 것처럼 보이게 만들 수 있습니다. 또한 스폰 위치, 이동 경로, 맵 전환 타이밍을 청크 단위로 제어하기 쉬운 장점이 있습니다.

### ADR-003. 인벤토리 구조

- 배경 : 아이템 종류와 기능이 계속 늘어나면서 비슷한 처리 로직이 여러 시스템에 분산되기 시작했습니다.
- 결정 : `InventoryManager`를 중심으로 기능별 하위 모듈을 연결하는 통합 인벤토리 구조를 채택했습니다.
- 채택 이유 : 아이템 추가, 차감, 조회, 이벤트 발행을 하나의 진입점에서 관리할 수 있고, 기능이 늘어나도 모듈 단위로 확장하기 쉽습니다. 중복 로직을 줄이고 여러 시스템에서 같은 아이템 데이터를 일관되게 다루기 위한 선택입니다.

### ADR-004. 데이터 저장 구조

- 배경 : Firebase를 도입해 서버 저장 구조까지 갖추기에는 프로젝트 일정이 부족했지만, 게임을 종료했다가 다시 실행해도 데이터가 유지되어야 했습니다.
- 결정 : 기능별 `SaveData`와 `JSONService`를 사용하는 로컬 JSON 저장 구조를 채택했습니다.
- 채택 이유 : JSON은 구조가 단순하고 다루기 쉬우며, 기능별 저장 데이터를 나눠 관리하기에도 적합했습니다. 구현과 디버깅 부담이 상대적으로 적고, 짧은 일정 안에서 안정적인 저장/복원 기능을 만드는 데 유리했습니다.

### ADR-005. UI와 씬 전환 프레임워크

- 배경 : UI 수가 많아질수록 화면마다 구현 방식이 달라질 수 있었고, 기능을 만들 때 공통 처리 단계를 빼먹을 가능성이 있었습니다.
- 결정 : UI 생명주기와 씬 전환 흐름을 공통 규칙으로 묶는 프레임워크를 채택했습니다.
- 채택 이유 : UI 초기화, 이벤트 구독, 화면 갱신, 해제 흐름을 통일하면 전체 화면의 일관성을 유지하기 좋습니다. 또한 기능을 추가할 때 필요한 단계를 정해진 순서로 구현하게 되어 누락을 줄일 수 있다고 판단했습니다.

### ADR-006. 런타임 최적화

- 배경 : 전투 중 많은 이펙트 오브젝트가 반복 생성되면서 성능 문제가 발생할 가능성이 높았습니다.
- 결정 : 반복 생성되는 오브젝트에 오브젝트 풀링을 적용했습니다.
- 채택 이유 : `Instantiate/Destroy` 호출을 줄이면 런타임 부하를 완화할 수 있고, 전투 상황이 복잡해져도 성능을 보다 안정적으로 유지할 수 있습니다. 특히 이펙트 사용량이 많은 프로젝트 특성상 선제적으로 대응할 필요가 있다고 판단했습니다.

## 핵심 흐름

- 데이터 로드 : `DataManager.cs`가 CSV 테이블을 메모리에 적재하고, 각 매니저가 필요한 초기 데이터를 가져와 런타임 상태를 구성합니다.
- 스테이지 진행 : `StageManager.cs`가 현재 진행 상태를 관리하고, `MapManager.cs`, `InfinityMap.cs`, `MonsterSpawner.cs`가 맵 구성과 전투 전개를 이어받습니다.
- 아이템 처리 : `InventoryManager.cs`가 아이템 변경 요청을 각 하위 모듈로 전달하고, UI와 게임 이벤트는 공통 이벤트 흐름을 통해 갱신됩니다.
- 저장 처리 : 각 시스템이 `SaveData`를 갱신하면 `AutoDataSaveManager.cs`가 Dirty 데이터를 모아 `JSONService.cs`로 저장합니다.
