# Gemini Visual Slide Pack

이 문서는 Unity 프로젝트 발표자료를 만들기 위한 `시각 중심 슬라이드 기획서`입니다.
중요한 점은 모든 기술에 똑같이 `Before/After`를 넣는 것이 아니라, 각 기술에 가장 잘 맞는 시각 자료 방식을 선택하는 것입니다.
아래 설명은 현재 프로젝트 코드에서 실제로 확인되는 구현만 기준으로 정리하며, 문서에 없는 기능은 추측하지 않습니다.

이 문서의 목적:
- 발표자료가 개발 문서처럼 보이지 않게 만들기
- 코드 설명보다 문제, 구조, 사용자 경험, 운영 효과를 중심으로 보여주기
- 실제 프로젝트 화면에 맞는 스크린샷, GIF, 플로우차트 아이디어를 바로 뽑아내기

핵심 원칙:
- 모든 슬라이드에 `Before/After`를 넣지 않는다.
- `전후 차이가 직관적인 기술`에만 Before/After를 사용한다.
- 나머지는 `실제 화면`, `플로우차트`, `구조도`, `운영 흐름 비교`로 설득한다.
- 각 슬라이드는 텍스트보다 시각 자료가 중심이 되어야 한다.
- 슬라이드 본문에는 파일명, 클래스명, 메서드명, 스크립트 경로를 쓰지 않는다.

시각 자료 선택 기준:
- 사용자 경험 차이가 큰 기술: Before/After 추천
- 내부 구조가 중요한 기술: 플로우차트 또는 구조도 추천
- 운영 효율이 중요한 기술: 프로세스 비교 또는 단계 축소 비교 추천
- 반복 동작이 중요한 기술: GIF 또는 연속 캡처 추천

## Slide 1. 스킬 모듈 정보 구조

### 슬라이드 핵심 메시지
하나의 스킬을 기본 정보와 코어 모듈, 확장 모듈로 나눠 관리하고, 이 구조가 현재 프로젝트의 스킬 UI와 전투 데이터에 연결되어 있다.

### 이 슬라이드에서 강조할 점
- 현재 프로젝트에서 스킬은 기본 정보와 `모듈 1~3`, 그리고 확장 슬롯에 연결되는 `모듈 4~5`로 구성된다는 점
- 확장 모듈 장착 상태가 UI 아이콘, 슬롯 개방 상태, 전투용 스킬 컨텍스트까지 이어진다는 점

### 추천 시각 방식
- 메인: 스킬 상세 패널 또는 스킬 리스트 화면
- 보조: `기본 스킬 정보 + M1/M2/M3 + M4/M5 확장 슬롯` 구조도
- 보조 2: 장착 가능한 확장 슬롯과 아이콘 상태 화면

### 이 기술에 Before/After가 꼭 필요한가?
- 필수는 아님
- 이 슬라이드는 Before/After보다 `실제 UI 화면 + 모듈 구조도`가 더 효과적임

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 스킬 상세 패널의 기본 정보 탭
- 스킬 상세 패널의 등급 정보 탭
- 확장 슬롯이 `잠김 / 장착 가능 / 장착 완료` 상태로 보이는 화면
- 확장 모듈 선택 팝업 또는 젬 선택 리스트 화면
- 스킬 리스트에서 레벨, 오픈 슬롯 수, 장착 가능 여부가 보이는 화면

### 추천 플로우차트
`스킬 선택 -> 기본 정보 확인 -> 모듈 1~3 핵심 구조 참조 -> 모듈 4/5 슬롯 개방 및 장착 상태 확인 -> 전투용 스킬 컨텍스트 구성`

### 필요할 때만 쓰는 비교 요소
`기본 스킬 정보만 설명` vs `모듈 구조와 확장 슬롯까지 함께 설명`

### 슬라이드 본문 예시
- 스킬 데이터는 기본 정보와 모듈 정보가 분리되어 구성됨
- 확장 슬롯은 잠김 여부와 장착 상태가 UI에서 바로 구분됨
- 장착된 확장 모듈 정보가 실제 전투용 스킬 구성까지 이어짐

### 발표자 설명 방향
이 슬라이드는 프리셋 자체보다 “스킬 하나가 어떤 정보 덩어리로 구성되어 있는가”를 보여주는 것이 핵심입니다. 현재 프로젝트에서는 기본 스킬 정보 위에 코어 모듈과 확장 모듈이 얹히는 구조이기 때문에, 상세 패널과 슬롯 상태 화면을 통해 모듈 정보가 어떻게 보이고 연결되는지를 설명하는 편이 가장 정확합니다.

## Slide 2. UI 해상도 대응 및 노치 대응

### 슬라이드 핵심 메시지
기기마다 달라지는 화면 예외를 수동 대응 대신 공통 Safe Area 구조로 자동 처리했다.

### 이 슬라이드에서 강조할 점
- 특정 기기에서 UI가 잘리던 문제를 구조적으로 해결했다는 점
- 씬 로드와 해상도 변화에도 다시 맞춰지는 공통 대응이라는 점

### 추천 시각 방식
- 메인: 노치 대응 전/후 비교
- 보조: Safe Area 적용 플로우차트
- 보조 2: 실제 게임 UI 화면

### 이 기술에 Before/After가 꼭 필요한가?
- 매우 추천
- 이 슬라이드는 전후 비교가 가장 직관적인 사례

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 상단 HUD가 노치에 닿는 화면과 보정 후 화면
- 하단 버튼 또는 메뉴 패널이 안전 영역 안으로 정리된 화면
- 씬 진입 직후 UI가 자동으로 재정렬된 화면
- 세로/가로 또는 해상도 차이가 보이는 두 장면

### 추천 플로우차트
`씬 진입 -> UI 루트 확인 -> 안전 영역 계산 -> 앵커 재배치 -> 노치 바깥 영역 마스킹 -> 최종 화면 반영`

### 추천 Before/After
| Before | After |
|---|---|
| 특정 기기에서 UI가 상단 노치에 겹침 | 안전 영역 안으로 자동 정렬 |
| 화면마다 수동 수정 필요 | 공통 구조에서 자동 재적용 |
| 기기별 완성도 편차 발생 | 다양한 해상도에서 일관된 UI 유지 |

### 슬라이드 본문 예시
- 안전 영역을 기준으로 UI 배치를 자동 계산
- 잘리는 영역은 마스킹해 시각 완성도 개선
- 씬 전환과 화면 변화에도 같은 규칙을 재사용

### 발표자 설명 방향
이 슬라이드는 기술 설명보다 결과 화면이 중요합니다. 같은 UI가 특정 기기에서는 어떻게 잘렸고, 지금은 어떻게 자연스럽게 들어오는지를 전후 비교로 보여주면 가장 설득력이 높습니다.

## Slide 3. CSV 기반 데이터 테이블 운영

### 슬라이드 핵심 메시지
데이터 파일 수정이 곧 게임 반영으로 이어지는 데이터 중심 작업 흐름을 만들었다.

### 이 슬라이드에서 강조할 점
- 밸런싱과 기획 데이터 수정이 빠르게 반복될 수 있었던 이유
- 코드 하드코딩이 아닌 테이블 기반 운영 구조라는 점

### 추천 시각 방식
- 메인: 데이터 시트 일부 + 게임 반영 화면
- 보조: 데이터 흐름 플로우차트
- 비교: 선택 사항, 공간이 남을 때만 작게 사용

### 이 기술에 Before/After가 꼭 필요한가?
- 필수 아님
- `원본 데이터 시트 -> 게임 반영` 흐름을 보여주는 편이 더 효과적임

### 현재 프로젝트에서 넣기 좋은 실제 화면
- CSV 또는 스프레드시트 일부 캡처
- 스킬 수치나 스테이지 값이 반영된 인게임 UI
- 보상, 몬스터 수치, 스킬 정보 같은 데이터가 화면에 나타나는 장면
- 로딩 성공 또는 데이터 반영 로그

### 추천 플로우차트
`데이터 시트 수정 -> 테이블 반영 -> 앱 로드 -> 데이터 파싱 -> 공통 데이터 캐시 구성 -> UI 및 게임 시스템 참조`

### 필요할 때만 쓰는 비교 요소
`코드 수정 중심 데이터 관리` vs `테이블 중심 운영`

### 슬라이드 본문 예시
- 데이터 수정이 곧 런타임 반영으로 이어지도록 흐름 구성
- 밸런스 수정과 검증 반복이 빠른 테이블 중심 운영
- 여러 시스템이 공통 데이터를 참조하도록 정리해 재사용성 확보

### 발표자 설명 방향
이 슬라이드는 “데이터를 수정하는 사람이 편해진 구조”를 보여주는 것이 좋습니다. 비교표를 크게 넣기보다, 데이터 시트가 게임 화면으로 이어지는 흐름도를 짧게 보여주는 쪽이 발표자료로 더 깔끔합니다.

## Slide 4. 어드레서블 업로드 자동화

### 슬라이드 핵심 메시지
원격 콘텐츠 배포를 수작업에서 자동 파이프라인으로 바꿔 운영 절차를 단순화했다.

### 이 슬라이드에서 강조할 점
- 플레이어가 직접 보지 않는 영역이지만 팀 운영 효율에 큰 영향을 주는 개선이라는 점
- 빌드, 업로드, 릴리즈 관리가 하나의 절차로 이어진다는 점

### 추천 시각 방식
- 메인: 운영 파이프라인 플로우차트
- 보조: 빌드 설정 화면 또는 업로드 결과 화면
- 비교: 전형적인 Before/After보다 `수동 절차 vs 자동 절차` 단계 비교가 더 적합

### 이 기술에 Before/After가 꼭 필요한가?
- 일반적인 화면 전후 비교는 비추천
- 대신 `절차 단계 비교`가 매우 효과적임

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 빌드 자동화 설정 화면
- 업로드 성공 로그 또는 배포 결과 화면
- 릴리즈나 배지 관리 결과가 보이는 화면
- 빌드 버튼 이후 자동 진행 순서를 정리한 다이어그램

### 추천 플로우차트
`빌드 실행 -> 배포 세션 준비 -> 콘텐츠 빌드 -> 원격 업로드 -> 릴리즈 반영 -> 설정 복구 및 로그 저장`

### 추천 절차 비교
| 기존 방식 | 자동화 이후 |
|---|---|
| 빌드 후 업로드를 별도로 수행 | 빌드 흐름 안에서 배포까지 자동 진행 |
| 설정과 업로드 단계를 사람이 직접 확인 | 세션과 로그 기준으로 일관되게 처리 |
| 반복 배포 시 누락 가능성 존재 | 절차 단순화로 실수 가능성 감소 |

### 슬라이드 본문 예시
- 빌드와 원격 콘텐츠 배포를 하나의 흐름으로 통합
- 릴리즈와 로그를 함께 관리해 운영 안정성 향상
- 반복 배포 시간을 줄이고 누락 가능성을 낮춤

### 발표자 설명 방향
이 슬라이드는 화면보다 절차가 핵심입니다. “예전에는 몇 단계를 사람이 직접 했고, 지금은 어떤 흐름으로 자동화됐는지”를 단계 수로 보여주면 발표에서 전달력이 좋습니다.

## Slide 5. JSON 기반 저장 시스템

### 슬라이드 핵심 메시지
저장 시스템을 단순 직렬화가 아니라 실제 플레이 복원과 안정성을 고려한 구조로 만들었다.

### 이 슬라이드에서 강조할 점
- 게임을 껐다 켜도 스킬 구성이나 진행 상태가 복원된다는 점
- 저장 중 문제가 생겨도 흐름이 완전히 깨지지 않도록 방어한 점

### 추천 시각 방식
- 메인: 저장/불러오기 흐름도
- 보조: 실제 저장 전후 게임 상태 화면
- 보조 2: 저장 파일 예시

### 이 기술에 Before/After가 꼭 필요한가?
- 필수 아님
- 이 슬라이드는 `복원 흐름`과 `실제 유지 화면`이 더 효과적임

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 스킬 프리셋이 저장된 상태 화면
- 스테이지 진행 상태가 유지된 화면
- 게임 재실행 후 동일 상태로 복원된 장면
- 저장된 JSON 파일 일부 또는 저장 폴더 화면

### 추천 플로우차트
`게임 상태 변경 -> 저장 데이터 반영 -> 임시 저장 -> 본 저장 파일 교체 -> 다음 실행 시 로드 -> 상태 복원`

### 필요할 때만 쓰는 비교 요소
`메모리 상태만 유지` vs `재실행 후 복원 가능한 저장 구조`

### 슬라이드 본문 예시
- 스킬 구성과 진행 상태를 파일 기반으로 안정적으로 저장
- 저장 과정에 안전 장치를 두어 손상 위험 완화
- 로드 실패 시 기본 상태를 제공해 플레이 흐름 방어

### 발표자 설명 방향
이 슬라이드는 기술 구조보다 “실제로 무엇이 저장되고 무엇이 다시 복원되는지”가 중요합니다. 저장 파일 화면만 보여주기보다, 저장 전과 재실행 후 상태가 유지되는 장면을 같이 넣는 편이 훨씬 좋습니다.

## Slide 6. 오브젝트 풀링

### 슬라이드 핵심 메시지
반복 생성되는 몬스터와 이펙트를 재사용 구조로 바꿔 전투 흐름을 더 안정적으로 관리했다.

### 이 슬라이드에서 강조할 점
- 반복 생성 비용을 줄였다는 점
- 단순 최적화가 아니라, 자주 등장하는 객체를 운영하기 쉬운 구조로 정리했다는 점

### 추천 시각 방식
- 메인: 전투 장면 또는 몬스터 스폰 장면
- 보조: 풀 라이프사이클 플로우차트
- 비교: Before/After 추천

### 이 기술에 Before/After가 꼭 필요한가?
- 추천
- 이 기술은 `반복 생성/삭제`와 `재사용 구조`의 차이를 비교하기 좋음

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 일반 몬스터가 여러 마리 스폰되는 장면
- 보스 몬스터 스폰 장면
- 파티클이 반복 재생되는 장면
- Hierarchy에서 풀 루트 아래 비활성 오브젝트가 쌓여 있는 화면

### 추천 플로우차트
`생성 요청 -> 재사용 가능한 오브젝트 확인 -> 없으면 생성 또는 로드 -> 사용 -> 비활성화 -> 풀로 반환`

### 추천 Before/After
| Before | After |
|---|---|
| 반복 생성과 삭제가 계속 발생 | 재사용 중심 구조 |
| 전투가 길어질수록 관리 비용 증가 | 자주 쓰는 객체를 안정적으로 운영 |
| 생성 방식마다 흐름이 나뉨 | 하나의 재사용 규칙으로 정리 |

### 슬라이드 본문 예시
- 몬스터와 이펙트를 반복 생성 대신 재사용 구조로 전환
- 재사용 시 초기화 규칙을 함께 두어 상태 꼬임 최소화
- 동적 로드 오브젝트까지 같은 흐름으로 연결

### 발표자 설명 방향
이 슬라이드는 체감 효과를 보여주기 좋습니다. 몬스터 스폰 장면과 풀 라이프사이클을 같이 보여주면 단순 성능 최적화가 아니라, 전투 구조를 정리한 작업이라는 점이 잘 드러납니다.

## Slide 7. 상태패턴 기반 행동 구조

### 슬라이드 핵심 메시지
복잡한 행동 분기를 상태 단위로 나눠 확장과 유지보수가 쉬운 구조로 정리했다.

### 이 슬라이드에서 강조할 점
- 기능이 늘어날수록 분기문이 아니라 상태 구조가 필요한 이유
- 플레이어와 적 모두 같은 전이 규칙으로 설명 가능한 점

### 추천 시각 방식
- 메인: 상태 전이 다이어그램
- 보조: 대표 상태 카드 3~4개
- 비교: 필요하면 작은 구조 비교만 사용

### 이 기술에 Before/After가 꼭 필요한가?
- 필수 아님
- 이 슬라이드는 전후 비교보다 `상태 전이 구조도`가 더 적합함

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 적의 `스폰 -> 추적 -> 공격 -> 사망` 흐름 다이어그램
- 플레이어 또는 적의 상태 변화가 보이는 연속 캡처
- 각 상태의 역할을 간단히 적은 카드형 시각 자료

### 추천 플로우차트
`이벤트/조건 발생 -> 다음 상태 요청 -> 현재 상태 종료 -> 다음 상태 진입 -> 상태별 업데이트 반복`

### 필요할 때만 쓰는 비교 요소
`거대한 분기문 중심 처리` vs `상태별 책임 분리`

### 슬라이드 본문 예시
- 행동을 진입, 업데이트, 종료 단위로 나눠 관리
- 공통 전이 규칙으로 여러 캐릭터 흐름을 일관되게 정리
- 상태별 책임이 분리되어 기능 추가와 디버깅이 쉬워짐

### 발표자 설명 방향
이 슬라이드는 화면 한 장보다 구조 그림이 더 중요합니다. 발표에서는 “왜 상태가 많아질수록 이런 구조가 필요했는지”를 짧게 설명하고, 실제 게임 흐름과 연결해서 보여주는 편이 좋습니다.

## Slide 8. NavMesh 기반 적 추적 및 사거리 판정

### 슬라이드 핵심 메시지
적의 추적과 공격 전환을 실제 플레이 감각에 맞게 다듬어 전투 흐름을 더 자연스럽게 만들었다.

### 이 슬라이드에서 강조할 점
- 단순히 플레이어를 향해 움직이는 것이 아니라, 실제 충돌 경계를 고려한 전환이라는 점
- 스폰부터 추적, 공격 전환까지 하나의 흐름으로 이어진다는 점

### 추천 시각 방식
- 메인: 적이 추적하는 실제 장면 GIF
- 보조: 추적 -> 공격 전환 플로우차트
- 비교: `중심 거리`와 `경계 거리` 비교 다이어그램 추천

### 이 기술에 Before/After가 꼭 필요한가?
- 일반적인 전후 비교보다는 `판정 방식 비교`가 더 효과적임

### 현재 프로젝트에서 넣기 좋은 실제 화면
- 몬스터가 플레이어를 추적하는 전투 장면
- 공격 범위에 진입하면서 행동이 바뀌는 장면
- 장애물이 있는 곳에서 추적이 이어지는 장면
- 보스 또는 큰 몬스터 기준으로 거리 판정이 자연스럽게 보이는 장면

### 추천 플로우차트
`몬스터 스폰 -> 추적 상태 진입 -> 플레이어 방향 경로 갱신 -> 공격 가능 거리 확인 -> 공격 상태 전환`

### 추천 비교 요소
| 비교 포인트 | 설명 |
|---|---|
| 중심 거리 기준 | 큰 캐릭터에서 체감이 어색할 수 있음 |
| 경계 거리 기준 | 실제 맞닿는 거리와 더 비슷한 판정 |

### 슬라이드 본문 예시
- 경로 탐색을 기반으로 장애물이 있는 전장에서도 자연스러운 추적 구현
- 공격 전환은 실제 충돌 경계를 기준으로 판단해 체감 정확도 개선
- 스폰 이후 추적 시작까지 포함해 반복 전투 흐름을 안정화

### 발표자 설명 방향
이 슬라이드는 AI 복잡도를 자랑하는 발표가 아니라, 플레이어가 느끼는 어색함을 줄인 사례처럼 보이게 만드는 것이 좋습니다. GIF와 간단한 비교 도식이 들어가면 설명이 훨씬 쉬워집니다.

### 추천 스타일
- 배경: 밝은 회색 또는 옅은 아이보리
- 포인트 색상: 네이비 + 청록, 또는 네이비 + 오렌지
- 텍스트 비중: 30%
- 시각 자료 비중: 70%

### 슬라이드 공통 규칙
- 각 슬라이드는 `문제 -> 해결 -> 효과`가 보이게 구성
- bullet은 3개 정도로 짧게 유지
- 플로우차트는 4~5단계 이내로 요약
- 비교 요소는 `효과적일 때만` 사용
- 실제 프로젝트 화면이 있으면 구조도보다 우선 사용

### 꼭 기억할 조건
- 슬라이드에 코드 파일명, 메서드명, 스크립트 경로를 쓰지 않는다
- 모든 기술을 같은 형식으로 억지로 맞추지 않는다
- 각 기술에 가장 잘 맞는 시각 자료 방식을 선택한다
- 발표자료가 기술 문서가 아니라 실제 프로젝트 포트폴리오처럼 보이게 만든다
